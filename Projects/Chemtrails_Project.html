Overview

The Problem

Travel planning is fragmented. Users rely on separate apps to search places, save ideas, plan schedules, and document experiences — making the process overwhelming and disconnected.

Fragmented
Planning across 6 apps
Users switch between Google Maps, Instagram, and Notes simultaneously — context-switching was the #1 frustration across all 5 interviews.
Lost saves
No consolidated view
4 of 5 participants forgot where they saved destinations because saves were scattered across different platforms with no single source of truth.
Impersonal
Camera roll ≠ memory
3 of 5 participants wanted to document memories that felt more personal than a camera roll — no context, no narrative, no story.
3+
Apps per trip
Every participant used at least 3 separate apps to plan a single trip — a baseline we used as our screening criteria for interviews.
Design Goals
🗺
Unify the planning experience
One app for search, save, schedule, and journal — eliminate context-switching entirely.
🧠
Reduce cognitive load
Soft tones, clear hierarchy, and generous spacing so the app feels like a travel companion, not a task manager.
📓
Make memory-keeping personal
Journal entries with inline photo uploads so trips become personal narratives, not unordered camera rolls.
🔒
Private, not social
A personal travel companion — not a platform for sharing. Every feature decision was filtered through this principle.

User Research

Who We
Designed For

5 semi-structured interviews with frequent travellers aged 22–35, each using at least 3 separate apps to plan a single trip.

Key Findings
01
Context-switching was the #1 pain
All 5 participants used Google Maps, Instagram, and Notes simultaneously — the mental overhead of switching between them was cited as more exhausting than the planning itself.
02
Saved places get lost
4 of 5 participants said they regularly forgot where they'd saved a destination — the save felt useful in the moment but was never actionable later.
03
Camera rolls feel impersonal
3 of 5 wanted a way to document memories with context and feeling — not just photos without narrative.

"The problem isn't planning — it's that my planning is scattered across 6 apps, so I can never see the full picture."

— Research Participant

Design Concept

Calm · Exploratory
· Personal

Chemtrails is designed to feel like a travel companion, not a task manager. Content — places, images, memories — is always the primary focus.

Design Principles
🌊
Calm and unhurried
The app should feel like browsing a travel magazine, not filling a spreadsheet. Soft tones and generous spacing reinforce this at every screen.
🖼
Content-first
UI chrome stays minimal so destination photos and journal memories take centre stage. The interface serves the content, never competes with it.
⚡
Low friction at every action
Saving a place, scheduling a visit, writing a journal entry — each should take under 30 seconds. Friction is removed, not hidden.

Technical Overview

Built End-to-End
in React Native

My first full mobile app built from scratch — navigation state, image uploads, API error handling, and calendar logic all required documentation-driven learning and iterative debugging.

⚛️
Frontend — React Native
Multi-screen navigation via bottom tab navigator. Custom components for place cards, journal entries, and calendar views. Real-time state management across all screens.
🔥
Backend — Firebase
Firebase Auth for secure accounts. Firestore real-time database for wishlist, journals, and user data. Firebase Storage for multi-photo journal uploads.
🗺
APIs — Google Places
Real-time destination search and autocomplete. Place details: ratings, reviews, photos, directions. API error handling and graceful fallback states throughout.
Challenges Overcome
01
Navigation state management
Managing shared state across the bottom tab navigator required a consistent data layer — solved with Firestore real-time listeners across all screens.
02
Inline image upload
Moving photo selection inline (rather than a separate screen) required custom file picker integration and progress state within the journal entry form.
03
Google Places API error handling
Network failures and empty results needed graceful fallback states so the app remained usable offline or with poor connectivity.
04
Calendar logic
Mapping wishlist items to calendar dates required custom date comparison and persistence logic — stored per-item in Firestore with real-time sync.

Reflection

What I Learned

Building a full mobile product end-to-end — research through deployment — taught me things that neither design nor development alone could have.

Cut ruthlessly
The social sharing feature prototyped well but contradicted what users actually wanted. Removing it was the right call. Every feature that doesn't serve the core use case weakens the product.
Test early, even with 3 people
Small tests surfaced the dashboard overwhelm issue and the hidden photo upload — both were critical fixes that would have shipped as problems otherwise.
Design and code in parallel
Building in React Native after designing in Figma created a productive feedback loop — implementation constraints pushed back on design in ways that improved both simultaneously.
Inline beats separate screen
Moving photo upload inline was a tiny structural change with an outsized effect on adoption. Reducing friction at the exact moment of action is the highest-leverage design move there is.
